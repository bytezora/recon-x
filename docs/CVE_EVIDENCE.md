# CVE Evidence Methodology

This document describes how to prove CVE precision and recall for recon-x without relying on marketing numbers.

## What Counts As Proof

A valid CVE accuracy claim needs all of the following:

- A fixed test scope: exact hosts, ports, products and versions.
- A ground-truth file: expected CVEs per host/port, backed by vendor advisories, NVD records, SBOM data or a pinned vulnerable lab image.
- A recon-x JSON scan produced with the exact command under test.
- An evidence report generated by `-cve-evidence`.
- Precision and recall both meeting the selected threshold, usually `0.90`.

The evidence report stores:

- true positives, false positives and false negatives
- precision, recall and F1
- scan file SHA-256
- ground-truth file SHA-256
- embedded CVE database hash
- tool version
- per-case TP/FP/FN lists

## Recommended 90% CVE Accuracy Run

Use strict matching and strong service evidence:

```bash
nmap -sV -oX nmap.xml lab.local
recon-x -target lab.local \
  -nmap-xml nmap.xml \
  -skip-portscan \
  -cve-live \
  -cve-profile strict \
  -cve-require-version \
  -json scan.json
```

Then evaluate the result:

```bash
recon-x \
  -cve-evidence ground-truth.json \
  -cve-evidence-scan scan.json \
  -cve-evidence-report evidence.json \
  -cve-evidence-markdown evidence.md \
  -cve-evidence-threshold 0.90
```

Exit codes:

- `0`: PASS, precision and recall are both at or above the threshold.
- `1`: command/input error.
- `2`: FAIL, at least one metric is below the threshold.

## Real-Domain Assurance Mode

For a normal domain you usually do not have ground truth. In that case, use `-cve-assurance` after producing `scan.json`:

```bash
nmap -sV -oX nmap.xml example.com
recon-x -target example.com \
  -nmap-xml nmap.xml \
  -skip-portscan \
  -cve-live \
  -cve-profile strict \
  -cve-require-version \
  -json scan.json

recon-x \
  -cve-assurance scan.json \
  -cve-assurance-report assurance.json \
  -cve-assurance-markdown assurance.md \
  -cve-assurance-threshold 0.90
```

Assurance mode is designed for authorized real domains. It does not invent a measured precision/recall number. Instead, it checks whether the scan has enough evidence to support a high-confidence public-service CVE claim:

- service fingerprints exist
- at least 90% have version evidence
- at least 90% have CPE evidence
- live NVD enrichment covers at least 90% of CPE fingerprints
- NVD query success is at least 95%
- CVE policy is strict and requires versions
- CVE findings are high or confirmed confidence

It always separates these two claims:

- Public-service service/version CVE assurance: may pass on a strong external scan.
- Whole-domain all-CVE 90% assurance: not provable from external unauthenticated scanning alone.

## Ground Truth Schema

```json
{
  "name": "local vulnerable lab",
  "source": "Docker image digests + vendor advisories + NVD references",
  "scope_unknown_as_fp": true,
  "ignore_cves": [],
  "cases": [
    {
      "name": "Apache httpd 2.4.49",
      "host": "lab.local",
      "port": 8080,
      "product": "apache",
      "version": "2.4.49",
      "expected_cves": ["CVE-2021-41773", "CVE-2021-42013"],
      "notes": "Pinned image digest sha256:..."
    }
  ]
}
```

`scope_unknown_as_fp` defaults to `true`. This is strict by design: any CVE reported outside the listed test cases is counted as a false positive unless you explicitly disable that behavior.

## Metric Definitions

- True positive: recon-x reported a CVE that exists in the ground truth for the same case.
- False positive: recon-x reported a CVE not present in the ground truth for that case.
- False negative: the ground truth contains a CVE that recon-x did not report.
- Precision: `TP / (TP + FP)`.
- Recall: `TP / (TP + FN)`.
- F1: harmonic mean of precision and recall.

## What This Does Not Prove

No external network scanner can prove "all CVEs in a domain" without enough visibility. External banners usually do not expose:

- OS package inventories
- backend libraries
- private services
- authenticated-only application features
- container image contents
- plugins and extensions hidden behind the application

To prove broader coverage, add authenticated scanning, SBOM/package inventory, internal host scans and image/package analysis to the same ground-truth process.
